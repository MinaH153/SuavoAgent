# SuavoAgent Schema Canary — Design Spec

**Date:** 2026-04-13
**Author:** Joshua Henein + Claude + Codex (3 adversarial review rounds)
**Repo:** github.com/MinaH153/SuavoAgent
**Branch:** main
**Status:** Approved design, pending implementation
**Review:** 28 findings from final Codex review + 4 from Claude self-review. All CRITICAL and HIGH issues resolved in Errata section.

---

## Problem

PMS vendors push updates that rename columns, change status GUIDs, add/remove tables, or alter relationships. SuavoAgent has NO mechanism to detect schema drift after a learned model or hand-built adapter is activated. Consequences:

- Learned adapter queries wrong table → wrong Rxs detected → wrong prescriptions dispatched
- PioneerRx adapter's status GUID lookup returns 0 matches → silent "no Rxs ready" when there are
- 3-tier query fallback in PioneerRxSqlEngine masks disappearing tables as graceful degradation
- `LearnedPmsAdapter.PullReadyAsync` swallows non-transient errors → returns empty list → missed deliveries
- `schema_epoch` and `schema_fingerprint` columns exist in `learning_session` but are never written — dead code

**Patient safety risk:** Wrong prescriptions delivered, or ready prescriptions never detected.

## Solution

A **schema canary** that verifies the detection contract on every polling cycle. Fails closed at the sync boundary — suspect data never reaches the cloud or drivers.

## Constraints

- Read-only SQL against PMS. Never modify PMS database.
- PHI never in canary artifacts. Only structural metadata (table/column names, types, status values).
- Must not increase SQL Server load beyond 100ms per 5-minute cycle.
- Must survive service restarts — drift hold state persisted in SQLite.
- Follows existing signed-command infrastructure for operator acknowledgment.
- Backward compatible — existing adapters work without canary until baseline is established.

## 1. Detection Contract & Baseline

Every active adapter declares a **detection contract** — the exact database objects, columns, CLR types, and status values it depends on. The contract is hashed into four component fingerprints plus a composite.

### PioneerRx Contract

```
Required Objects (with CLR types):
  Prescription.RxTransaction:
    RxTransactionID            uniqueidentifier
    DateFilled                 datetime
    DispensedQuantity          decimal
    RxTransactionStatusTypeID  uniqueidentifier
    RxID                       uniqueidentifier
    DispensedItemID            uniqueidentifier

  Prescription.Rx:
    RxID                       uniqueidentifier
    RxNumber                   int

  Prescription.RxTransactionStatusType:
    RxTransactionStatusTypeID  uniqueidentifier
    Description                nvarchar

  Inventory.Item:
    ItemID                     uniqueidentifier
    ItemName                   nvarchar
    NDC                        nvarchar

Required Status Map (sorted, exact 3-of-3):
  "To Be Put in Bin"       → {GUID}
  "Waiting for Delivery"   → {GUID}
  "Waiting for Pick up"    → {GUID}

Result Shape (ReadReadyMetadataAsync):
  RxNumber          int
  DateFilled        datetime
  DispensedQuantity decimal
  TradeName         nvarchar
  NDC               nvarchar
  StatusGuid        uniqueidentifier
```

### Learned Adapter Contract

Generated by `AdapterGenerator.BuildDetectionQuery`:

```
Required Objects:
  {primaryTable}: {rxNumberColumn}, {statusColumn}, {dateColumn?}

Required Status Values:
  @s0 = "value0", @s1 = "value1", ...

Approved Query: exact parameterized SQL
Result Shape: consumed column names + DataTypeNames
```

### Four Component Hashes

| Hash | Covers | Drift means |
|------|--------|-------------|
| `object_fingerprint` | Tables + columns + CLR types + nullability for all contracted objects | Table/column renamed, type changed, column dropped |
| `status_map_fingerprint` | Sorted (Description, GUID) tuples from status lookup table | Status renamed, GUID changed, count mismatch |
| `query_fingerprint` | Approved SQL text | Query structure modified |
| `result_shape_fingerprint` | Expected column names + DataTypeNames in query result set | Query returns different shape |

Composite: `contract_fingerprint = SHA-256(object + status + query + result)`

Component hashes tell you WHAT drifted, not just THAT something drifted.

### Baseline Storage

New `schema_canary_baselines` table in AgentStateDb (SQLite), keyed by `(pharmacy_id, adapter_type)`. Baseline set once during adapter activation. Re-verified every detection cycle.

## 2. Canary Execution — Preflight & Postflight

Every detection cycle runs two canary checkpoints on the **same SqlConnection instance**.

### Preflight (before query execution)

1. Query `sys.objects` / `sys.columns` / `sys.types` for ONLY contracted objects (4 tables, ~15 columns). `ORDER BY` schema, table, column for deterministic hashing.
2. Hash → fresh `object_fingerprint`
3. Query `Prescription.RxTransactionStatusType` for delivery-ready descriptions. `ORDER BY Description, RxTransactionStatusTypeID`.
4. Hash → fresh `status_map_fingerprint`
5. Compare both against stored baseline
6. Mismatch → classify severity → respond per tier
7. Match → proceed to detection query

### Connection Guard

If `SqlConnection.State` changes between preflight and detection query → abort cycle → rerun preflight on new session. Prevents stale preflight fingerprint on a reconnected session.

### Postflight (after ExecuteReaderAsync, BEFORE iterating rows)

1. Call `reader.GetColumnSchema()` immediately after `ExecuteReaderAsync`
2. Hash `ColumnName` + `DataTypeName` for each column → `result_shape_fingerprint`
3. Compare against baseline
4. Mismatch → batch tainted → drop it → drift_hold
5. Match → iterate rows, build batch

**Critical:** Postflight MUST run before any `reader.ReadAsync()` call. The existing `GetStringOrDefault` / `GetGuidOrDefault` helpers return `""`, `0`, `Guid.Empty` on missing columns — they mask drift as valid data. Postflight blocks this.

### Performance

One `sys.*` query (~15 rows) + one status query (~20 rows) + one `GetColumnSchema()` call. Total overhead <100ms per 5-minute cycle. The status query replaces the existing once-on-connect cache (net new cost is only the `sys.*` hit).

### Execution Flow

```
Detection Cycle Start
  │
  1. Acquire SqlConnection (reuse or open)
  2. PREFLIGHT
     a. sys.objects/sys.columns/sys.types for contracted objects
     b. Hash → object_fingerprint
     c. RxTransactionStatusType query (ORDER BY)
     d. Hash → status_map_fingerprint
     e. Compare against baseline
     f. MISMATCH → classify → respond
     g. MATCH → continue
  │
  3. CONNECTION GUARD — if state changed, abort, rerun from 1
  │
  4. EXECUTE detection query
  │
  5. POSTFLIGHT (before reading rows)
     a. reader.GetColumnSchema()
     b. Hash ColumnName + DataTypeName → result_shape_fingerprint
     c. Compare against baseline
     d. MISMATCH → drop batch → drift_hold
     e. MATCH → iterate rows, build batch
  │
  6. SYNC to cloud (only if all checks passed)
```

## 3. Severity Classification & Response

### Three Tiers

| Tier | Triggers | Response |
|------|----------|----------|
| **Advisory** | Change on non-contracted object (new unrelated table, new column on table we don't query) | Log. Continue. No hold. |
| **Warning** | Column type widened (same base type, larger size). Nullable flag changed. Column ordinal changed. | Reprobe immediately on fresh connection. If second probe confirms → escalate to Critical. If clean → clear, continue. |
| **Critical** | Required table missing. Required column missing. Column type changed incompatibly. Status description renamed. Status GUID changed. Status count mismatch. Result shape mismatch. Tier downgrade. | **Drop batch. Enter drift_hold. Increment schema_epoch.** |

### Classification Logic

```
IF any required table missing from sys.objects        → CRITICAL
IF any required column missing from sys.columns       → CRITICAL
IF any required column type changed incompatibly      → CRITICAL
IF status description count ≠ baseline count          → CRITICAL
IF any status description string changed              → CRITICAL
IF any status GUID changed for same description       → CRITICAL
IF result_shape_fingerprint mismatch                  → CRITICAL
IF tier downgrade detected                            → CRITICAL
IF duplicate status descriptions detected             → CRITICAL

IF column type widened (same base type, larger size)  → WARNING
IF nullable flag changed                              → WARNING
IF column ordinal changed (same name/type)            → WARNING

IF change is on non-contracted object                 → ADVISORY
```

Extracted as a pure function: `SchemaCanaryClassifier.Classify(ContractBaseline, ObservedContract) → CanarySeverity`

### Drift Hold Behavior

- Detection cycles continue but NEVER sync to cloud
- Each cycle re-runs preflight to check if drift resolved
- Dropped batches logged to `schema_canary_incidents` (not `unsynced_batches`)
- Suspect data never enters the retry queue
- Heartbeat frequency increases to 10s locally (faster operator feedback)

### Escalation Ladder

| Blocked cycles | Elapsed | Action |
|---------------|---------|--------|
| 1 | 0 min | Incident logged locally + heartbeat telemetry |
| 3 | ~15 min | Dashboard alert (amber badge on Agent Console) |
| 12 | ~1 hour | Heartbeat reports drift_hold → cloud triggers phone/SMS escalation |
| 72 | ~6 hours | Agent logs "Extended drift_hold — manual intervention required" |

### Resume Rules

- **Advisory:** Auto-clears. Never enters drift_hold.
- **Warning:** 2 consecutive clean probes → auto-clear.
- **Critical:** Requires operator acknowledgment via signed `acknowledge_drift` command. Human must verify the new schema.

## 4. Storage & Interface Changes

### New Interface: ICanaryDetectionSource

`RxDetectionWorker` uses `PioneerRxSqlEngine` directly — it does NOT go through `ILocalPmsAdapter`. The canary needs a new shared abstraction both paths implement.

```csharp
// SuavoAgent.Contracts/Canary/ICanaryDetectionSource.cs
public interface ICanaryDetectionSource
{
    string AdapterType { get; }
    ContractBaseline GetContractBaseline();
    Task<ContractVerification> VerifyPreflightAsync(
        ContractBaseline approved, CancellationToken ct);
    Task<DetectionResult> DetectWithCanaryAsync(
        ContractBaseline approved, CancellationToken ct);
}

public record DetectionResult(
    IReadOnlyList<RxMetadata> Rxs,
    ContractVerification PostflightVerification);
```

`ILocalPmsAdapter` stays unchanged — it's for capability discovery and writeback, not detection.

### New Types (in SuavoAgent.Contracts)

```csharp
public record ContractBaseline(
    string AdapterType,
    string ObjectFingerprint,
    string StatusMapFingerprint,
    string QueryFingerprint,
    string ResultShapeFingerprint,
    string ContractFingerprint,
    string ContractJson,
    int SchemaEpoch);

public record ContractVerification(
    bool IsValid,
    CanarySeverity Severity,
    IReadOnlyList<string> DriftedComponents,
    string? BaselineHash,
    string? ObservedHash,
    string? Details);

public enum CanarySeverity { None, Advisory, Warning, Critical }
```

### New Types (in SuavoAgent.Core)

- `SchemaCanaryClassifier` — pure function: `Classify(baseline, observed) → CanarySeverity`
- `SchemaCanaryEscalation` — pure state machine: `Transition(holdState, severity) → holdState`
- `PioneerRxCanarySource : ICanaryDetectionSource` — wraps PioneerRxSqlEngine
- `LearnedCanarySource : ICanaryDetectionSource` — wraps LearnedPmsAdapter

## 5. Heartbeat Integration & Cloud Reporting

### Fix: Signed Command Delivery (existing bug)

The current `approve_pom` flow stores `pending_pom_approval` in `agent_instances.config_json`, but the heartbeat route never reads it back or returns it in `data.signedCommand`. **Signed command delivery is broken.**

Fix: New cloud-side table:

```sql
CREATE TABLE agent_pending_commands (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_id UUID NOT NULL REFERENCES agent_instances(id),
    command_type TEXT NOT NULL,
    signed_envelope JSONB NOT NULL,
    incident_id TEXT,
    status TEXT NOT NULL DEFAULT 'pending'
        CHECK (status IN ('pending', 'delivered', 'consumed', 'expired')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    delivered_at TIMESTAMPTZ,
    expires_at TIMESTAMPTZ NOT NULL DEFAULT NOW() + INTERVAL '1 hour'
);
```

Heartbeat route loads oldest pending command, returns in `data.signedCommand`, marks `delivered`. Agent confirms consumption on next heartbeat.

### Heartbeat Payload

```json
{
  "canary": {
    "status": "clean | drift_hold",
    "severity": "none | advisory | warning | critical",
    "schemaEpoch": 1,
    "contractFingerprint": "a3f8...",
    "incidentId": "inc-20260413-001",
    "driftedComponents": ["status_map"],
    "expected": { "status_map": "b7e2..." },
    "observed": { "status_map": "d4a1..." },
    "blockedCycles": 3,
    "driftHoldSince": "2026-04-13T10:15:00Z",
    "lastVerifiedAt": "2026-04-13T10:30:00Z"
  }
}
```

Structured data, not human-readable string. Dashboard renders programmatically.

### Cloud Storage

Canary status stored in `pharmacy_data_snapshots` as `snapshot_type='canary_status'`. NOT in `config_json` (already bloated with api_key_hash, stats, observation fields, instincts).

### Dashboard

Agent Console canary indicator: green (clean), amber (warning), red (drift_hold). `acknowledge_drift` button triggers signed command through `agent_pending_commands`.

### Heartbeat Frequency

Agent locally shortens heartbeat to 10s during `drift_hold` (faster operator feedback). Reverts to 30s after resolution. No cloud flag needed.

### HealthSnapshot

Add canary state to shared health model used by both heartbeat and IPC `get_health`. Future system tray app reads canary status via IPC.

### acknowledge_drift Command

Two modes:

**A) Resume supervised (old baseline):**
```json
{ "action": "resume_supervised", "incidentId": "inc-..." }
```
Agent resumes detection. All batches require manual approval. Old baseline unchanged.

**B) Approve new baseline (requires review):**

Flow:
1. Agent uploads canary diff artifact to `POST /api/agent/canary-diff`
2. Operator reviews diff in dashboard
3. Operator approves → cloud signs command with reviewed artifact digest:
```json
{
  "action": "approve_new_baseline",
  "incidentId": "inc-...",
  "approvedDiffDigest": "sha256-of-reviewed-diff",
  "targetSchemaEpoch": 2
}
```
4. Agent re-hashes current state, verifies match, commits new baseline, increments epoch, resumes

Same TOCTOU protection as existing POM approval.

## Review Errata (Final Gate — Codex + Claude)

28 findings from Codex final review + 4 from Claude self-review. Resolutions below override the main spec sections where they conflict.

### CRITICAL Fixes

**E1. Baseline establishment deadlock (Codex #2).**
Spec says "first run, no baseline → fails closed" AND "baseline set during adapter activation." These contradict — agent can never start.
**Resolution:** On first detection cycle with no baseline, the canary runs preflight in **establishment mode**: executes all preflight queries, captures the result as the initial baseline, persists it, and logs a `baseline_established` audit entry. The first cycle's batch is synced normally (the baseline IS the current schema — no drift possible). Subsequent cycles verify against this baseline. No signed command needed for initial establishment. Establishment mode is a one-time transition, never re-entered unless the baseline is explicitly deleted.

**E2. ICanaryDetectionSource DI wiring (Codex #1).**
Spec introduces the interface but doesn't describe how `RxDetectionWorker` gets it.
**Resolution:** `RxDetectionWorker` receives `ICanaryDetectionSource?` via constructor injection (nullable — backward compatible). In `Program.cs`, if `LearningMode=false`, register `PioneerRxCanarySource` as `ICanaryDetectionSource`. If `LearningMode=true` AND phase is `active`, register `LearnedCanarySource`. If learning is still in progress (phase < active), don't register — canary is not active during learning. Worker checks `_canarySource != null` before running canary gates; if null, falls back to current behavior (no canary).

**E3. ReadReadyMetadataAsync LEFT JOIN vs. required contract (Codex #3).**
`BuildMetadataQuery` uses `LEFT JOIN Inventory.Item`. If Item table is missing, the query still returns rows with NULL TradeName/NDC — not a SQL error. But the canary contract marks Item as required, so preflight would fire Critical for pharmacies where Item doesn't exist.
**Resolution:** Split the contract into `required_objects` and `optional_objects`. `Inventory.Item` is optional. Preflight: missing required object → Critical. Missing optional object → Warning (logged, batch still proceeds but operator notified that drug names will be missing). Postflight: if Item is optional and missing, result shape is allowed to omit TradeName/NDC columns. Update the result shape baseline to have a "with-Item" and "without-Item" variant, or hash only required columns in the result shape.

**E4. Hold table PK (Codex #4).**
`schema_canary_hold` uses `pharmacy_id TEXT PRIMARY KEY` but baselines are keyed `(pharmacy_id, adapter_type)`.
**Resolution:** Change PK to `PRIMARY KEY (pharmacy_id, adapter_type)`. Drop the single-column PK.

**E5. Signed command delivery mechanism (Codex #5).**
The spec identifies the broken `approve_pom` delivery but the existing `HandleApprovePomAsync` in HeartbeatWorker IS processing commands (it works). Investigation: the heartbeat route in the `Suavo/src` checkout (not `web/web`) DOES deliver signed commands via `data.signedCommand`. The `web/web` checkout may be a stale/incomplete copy. 
**Resolution:** Verify which checkout is deployed. If `Suavo/src` is canonical, the delivery mechanism works and `agent_pending_commands` is an improvement but not a prerequisite. If `web/web` is canonical, implement `agent_pending_commands` as specified. Either way, `acknowledge_drift` uses the same delivery path as `approve_pom`.

### HIGH Fixes

**E6. LearnedPmsAdapter error handling (Codex #6).**
Spec says "Return ContractVerification instead of swallowing errors" on `LearnedPmsAdapter.cs`. Ambiguous whether this changes `PullReadyAsync` (which would break `ILocalPmsAdapter`).
**Resolution:** `LearnedPmsAdapter.PullReadyAsync` remains unchanged — it still implements `ILocalPmsAdapter`. The `LearnedCanarySource` wrapper catches the swallowed errors by detecting empty results + checking adapter health, and converts to `ContractVerification`. The wrapper also adds pre-query preflight and post-query postflight. `LearnedPmsAdapter` itself is modified only to expose its connection state and last error (new internal properties), not to change its interface.

**E7. Status GUID cache replacement (Codex #7).**
Spec says re-query per cycle but doesn't address the existing `_deliveryReadyGuids` cache.
**Resolution:** `PioneerRxSqlEngine._deliveryReadyGuids` cache is kept for `ReadReadyPrescriptionsAsync` (the full query path, used for on-demand patient fetch). The canary preflight queries status independently every cycle and stores the result in the canary verification, NOT in `_deliveryReadyGuids`. Two separate concerns: detection canary (per-cycle fresh query) vs. operational cache (per-connection, used by non-canary paths).

**E8. schema_epoch read in GetLearningSession (Codex #8).**
`schema_epoch` IS read at line 638 even though never written.
**Resolution:** Keep the column. Don't deprecate. Wire up `IncrementSchemaEpoch()` in the canary (already planned). Remove only `schema_fingerprint` which is truly never read or written.

**E9. Connection guard mechanism (Codex #9).**
"If SqlConnection.State changes" is underspecified.
**Resolution:** Snapshot `_connection.ClientConnectionId` (a Guid unique per physical connection) before preflight. After preflight completes, compare again before executing detection query. If different → new physical connection → abort cycle. This is more reliable than checking `State` (which can be `Open` on a different connection after reconnect). Fallback for providers that don't support `ClientConnectionId`: wrap preflight + query + postflight in a try-catch; any `SqlException` with a connection-class error number aborts the cycle.

**E10. "Resume supervised" manual approval mechanism (Codex #10).**
"All batches require manual approval" in supervised mode has no implementation path for detection batches.
**Resolution:** "Resume supervised" means: detection continues, batches sync to cloud, but every batch is tagged `requires_verification: true` in the sync payload. The fleet operator dashboard shows these batches with an amber indicator and must click "Verify" before dispatching drivers. This is a cloud-side/dashboard change — the agent just tags the payload. No per-batch signed command needed.

**E11. Heartbeat rate limit during drift_hold (Codex #11).**
10s heartbeat could hit cloud rate limits.
**Resolution:** Increase to 15s during drift_hold (not 10s). Cloud heartbeat endpoint rate limit is per-agent, currently generous (no explicit limit in code). Add a note to verify cloud-side rate limits during implementation. If rate-limited, agent falls back to 30s with a log warning.

### MEDIUM Fixes (selected)

**E12. Advisory tier unreachable (Codex #28).**
Preflight only queries contracted objects — can never see non-contracted changes.
**Resolution:** Remove Advisory tier entirely. It was aspirational and adds complexity with zero value if the preflight scope is contracted-only. Two tiers: Warning and Critical. Simplify `CanarySeverity` to `{ None, Warning, Critical }`.

**E13. Canary diff artifact format (Codex #14).**
Spec doesn't define payload format for `POST /api/agent/canary-diff`.
**Resolution:** JSON payload: `{ incidentId, pharmacyId, adapterType, schemaEpoch, baseline: { objectFingerprint, statusMap: [{description, guid}], resultShape: [{name, type}] }, observed: { same structure }, driftedComponents: ["status_map"], details: "human-readable summary" }`. Authenticated via HMAC (`verifyAgentRequest`). Max 500KB.

**E14. NULL baseline_contract_fingerprint on existing batches (Codex #15).**
Existing `unsynced_batches` rows have NULL for new columns.
**Resolution:** NULL fingerprint = pre-canary batch. During drift_hold, pre-canary batches are still retried (they were created before canary existed, likely valid). Only batches with a non-NULL fingerprint that doesn't match current baseline are skipped.

**E15. Contract versioning (Claude #D).**
No version field on contracts — future contract changes break old baselines.
**Resolution:** Add `contract_version INTEGER NOT NULL DEFAULT 1` to `schema_canary_baselines`. If agent upgrades and contract version changes, the baseline is invalidated and re-established (same as E1 first-run flow).

**E16. LearnedCanarySource type conversion (Claude #B).**
`LearnedPmsAdapter.PullReadyAsync` returns `RxReadyForDelivery`, not `RxMetadata`. `DetectionResult` uses `RxMetadata`.
**Resolution:** `LearnedCanarySource.DetectWithCanaryAsync` converts `RxReadyForDelivery` → `RxMetadata` (drop patient fields, keep RxNumber/DrugName/NDC/DateFilled/Quantity/StatusGuid/DetectedAt). This is a safe narrowing — `RxMetadata` is the PHI-free subset.

**E17. Learning mode canary scope (Claude #C).**
Spec doesn't address canary during learning phase.
**Resolution:** Canary is NOT active during learning (phases discovery/pattern/model). Learning is inherently exploratory — it's discovering the schema, not relying on a known one. Canary activates only when an adapter is activated (phase = active or PioneerRx hand-built mode). Explicitly stated in E2.

**E18. PioneerRxCanarySource assembly location (Codex #17).**
`PioneerRxCanarySource` in Core creates upward dependency on Adapters.PioneerRx.
**Resolution:** Move `PioneerRxCanarySource` to `SuavoAgent.Adapters.PioneerRx` assembly. Register it in `Program.cs` via DI. Core depends on `ICanaryDetectionSource` (in Contracts), not on the concrete implementation.

**E19. query_fingerprint instability for PioneerRx (Codex #24).**
`BuildMetadataQuery` generates SQL dynamically based on status count. If status count changes, query text changes, query_fingerprint changes → false Critical.
**Resolution:** For the PioneerRx hand-built adapter, `query_fingerprint` hashes the query TEMPLATE (with `{statusParams}` placeholder), not the fully-expanded SQL. The status values are covered by `status_map_fingerprint`. This separates "query structure changed" from "status list changed."

**E20. Warning tier — should it block detection? (Codex #18).**
Two clean probes = 10 minutes blocked for a potentially transient issue.
**Resolution:** Warning does NOT block detection. Batches proceed normally. Warning only: logs the anomaly, increments a warning counter in heartbeat telemetry, and triggers a reprobe next cycle. If the warning persists for 3 consecutive cycles, escalate to Critical (which does block). This prevents 10-minute false-positive blocks for transient DDL operations.
